## Summary

Phase 3 is a **port-and-wire phase** — not a design or library-selection phase. Every visual element, interaction rule, CSS value, and prose string is already specified in the design handoff (`design_handoff_explain_that_inchi/app.jsx`, `layers-info.js`, `styles.css`). The planner's job is to sequence tasks that translate the vanilla-JS handoff into TypeScript React components that consume Zustand store state.

The phase has three distinct tracks of work: (1) **data layer extension** — adding `atomElements` parsing to `parseAuxMapping.ts` and wiring it through the store and `App.tsx`; (2) **library module** — porting `layers-info.js` to `src/lib/layerInfo.ts` with full TypeScript types; (3) **component authoring** — implementing `InchiSection`, `LayerText`, `Explanation`, and `Legend` as CSS Module React components.

The design tokens in `src/styles.css` are already complete — including all layer accent colours, sub-token colours, typography variables, and spacing. No new global CSS is needed. The only global CSS already present covers `inchi-section`, `inchi-display`, `inchi-layer`, `explain`, `card`, `legend-row`, and `legend-tip` selectors. These exist in `src/styles.css` from the handoff but are not yet consumed by any React component.

**Primary recommendation:** Follow the handoff faithfully. Do not redesign; do not add abstraction. The only permitted deviations are (a) converting global class names to CSS Module camelCase, (b) replacing `mol.atoms` lookups with `atomElements` from the store, and (c) converting vanilla-React state (`useState`) to Zustand store reads.

### Pattern 1: Store Extension with Backward Compatibility

**What:** `atomElements: Record<number, string>` is a 1-based index → element symbol map (e.g., `{1: 'C', 2: 'C', 3: 'N'}`). It is derived by parsing the `rA:` field in the AuxInfo block.

**Source format (from real Ketcher 3.12.0 benzene output):**
`rA:6CCCCCC` — the digit after `rA:` is atom count, followed by element symbols in canonical order (one char each for single-letter elements; the format concatenates them directly).

**Parsing approach — extract the rA: field:**
```typescript
// Source: CONTEXT.md specifics + verified against BENZENE_AUXINFO_BODY in parseAuxMapping.test.ts
function parseAtomElements(auxBody: string): Record<number, string> {
  const parts = auxBody.split('/');
  const raPart = parts.find(p => p.startsWith('rA:'));
  if (!raPart) return {};
  // Format: "rA:6CCCCCC" — skip "rA:", read digit(s) for count, then symbols
  const body = raPart.slice(3);            // "6CCCCCC"
  const countMatch = body.match(/^(\d+)/);
  if (!countMatch) return {};
  const count = parseInt(countMatch[1], 10);
  const symbols = body.slice(countMatch[1].length); // "CCCCCC"
  const map: Record<number, string> = {};
  // Each element symbol is one or two characters (Cl, Br, etc.).
  // The rA: field uses single characters per atom — multi-letter elements
  // are encoded as two characters. Parse greedily: uppercase followed by optional lowercase.
  let i = 0, canon = 1;
  while (i < symbols.length && canon <= count) {
    const twoChar = symbols.slice(i, i + 2);
    const oneChar = symbols[i];
    // Check for two-character element (Cl, Br, etc.)
    if (i + 1 < symbols.length && /[a-z]/.test(symbols[i + 1])) {
      map[canon] = twoChar;
      i += 2;
    } else {
      map[canon] = oneChar;
      i += 1;
    }
    canon++;
  }
  return map;
}
```

**IMPORTANT NOTE:** The `rA:` field in the benzene test fixture is `rA:6CCCCCC` — all single-character elements. Multi-letter elements (Cl, Br) would be e.g. `ClBr` — two chars each. The greedy parse above handles both. [VERIFIED: BENZENE_AUXINFO_BODY in parseAuxMapping.test.ts line 6]

### Pattern 2: CSS Module with `data-layer` Attribute Selectors

**What:** The handoff uses CSS descendant selectors like `.inchi-layer[data-layer="formula"] .el-C { color: ... }`. In CSS Modules, the `data-layer` attribute is a real HTML attribute (not a class), so attribute selectors work without `:global`.

**How it works:** Vite CSS Modules scope class names (`.elC` → `.InchiSection_elC__xyz`) but do NOT scope attribute selectors. So `[data-layer="formula"] .elC { color: var(--c-el-C) }` inside a module file works correctly — the attribute selector is unscoped, but `.elC` is scoped to the component. This means `<span className={styles.elC}>` combined with `data-layer="formula"` on the parent span will match.

**Example from handoff ported to module syntax:**
```css
/* InchiSection.module.css */
.inchiLayer {
  display: inline;
  padding: 2px 4px;
  border-radius: 3px;
  cursor: pointer;
  transition: background 160ms ease, color 160ms ease;
}
.active { /* applied when hoverIdx === i */ }
.dim { opacity: 0.35; }

/* Attribute selectors work without :global */
[data-layer="formula"] .elC { color: var(--c-el-C); }
[data-layer="formula"] .elH { color: var(--c-el-H); }
/* ...etc */
[data-layer="t"] .parityPlus,
[data-layer="b"] .parityPlus { color: var(--c-stereo-plus); }
```

## Common Pitfalls

### Pitfall 1: `el-${el}` Dynamic Class Names Don't Work in CSS Modules

**What goes wrong:** The handoff uses `className={"el-" + el + " inchi-subtoken"}`. In CSS Modules this produces the literal string `"el-C inchi-subtoken"` which does not match the scoped class name `.InchiSection_elC__xyz`.

**Why it happens:** CSS Modules replace class names with scoped identifiers. Dynamic string construction bypasses the module lookup.

**How to avoid:** Use a static lookup object inside the component:
```typescript
const EL_CLASS: Record<string, string> = {
  C: styles.elC, H: styles.elH, N: styles.elN, O: styles.elO,
  S: styles.elS, P: styles.elP, F: styles.elF, Cl: styles.elCl,
  Br: styles.elBr, I: styles.elI,
};
// In FormulaText:
className={`${EL_CLASS[el] ?? ''} ${styles.inchiSubtoken}`}
```

**Warning signs:** Elements render in the default formula color instead of per-element colors.

### Pitfall 2: `setInchiData` Signature Change Breaks Existing Tests

**What goes wrong:** Adding a fourth `atomElements` argument to `setInchiData` is a breaking change. The existing store test calls `setInchiData` with 3 args (line 35) and the `beforeEach` `setState` call does not include `atomElements` (line 10).

**Why it happens:** TypeScript strict mode will cause a compile error if the 4th argument is required and not provided. If it's optional (`atomElements?:`), tests pass but App.tsx may forget to pass it.

**How to avoid:** Make `atomElements` required in the interface. Update `store.test.ts` to pass `{}` as the fourth argument in the `setInchiData` call, and add `atomElements: {}` to the `beforeEach` `setState` reset. Run `npm test` immediately after the store change to confirm.

**Warning signs:** TypeScript error TS2554 "Expected 4 arguments, but got 3" in test or App.tsx.

### Pitfall 3: `--accent` Custom Property Not Inherited by `::before`

**What goes wrong:** Setting `--accent` on the `.card` element via inline style might not cascade to the `::before` pseudo-element in some rendering contexts.

**Why it happens:** CSS custom properties DO cascade to pseudo-elements — this is standard behavior. However, if the inline style sets `style={{ '--accent': 'var(--ink-faint)' }}` in idle state and `style={{ '--accent': 'var(--c-conn)' }}` in active state, the transition on `::before` only works if the background is always `var(--accent, var(--line))`.

**How to avoid:** Always set `--accent` on the card — including idle state — using `var(--ink-faint)` as the idle value. The `::before` rule `background: var(--accent, var(--line))` then transitions smoothly. Do not conditionally omit the inline style.

### Pitfall 4: Legend Tooltip Clips at Card Edge

**What goes wrong:** The `.legend-tip` tooltip has `position: absolute; right: calc(100% + 12px)` — it extends outside the card. If the parent `.card.legend-card` has `overflow: hidden`, the tooltip is clipped.

**Why it happens:** The handoff CSS includes `.explain .legend-card { overflow: visible; }` to override the card's default. CSS Modules require this override to be in the same module or cascade correctly.

**How to avoid:** In `Explanation.module.css`, ensure the legend card class sets `overflow: visible` explicitly. The handoff source `.explain .legend-card { overflow: visible; }` confirms this is required. [VERIFIED: src/styles.css line 526]

### Pitfall 5: `parseMobileHydrogens` Already Exported — Don't Duplicate

**What goes wrong:** Porting `readingFor` from `layers-info.js` includes an inline `parseMobileHydrogens` reimplementation.

**Why it happens:** The handoff JS file defines it as a local function. The TypeScript codebase already exports it from `parseInchi.ts`.

**How to avoid:** In `layerInfo.ts`, import `parseMobileHydrogens`, `parseConnectionBonds`, and `parseStereoAtoms` from `./parseInchi` rather than re-defining them.

### Pitfall 6: `rA:` Field Atom Count Mismatch for Multi-Character Elements

**What goes wrong:** The `rA:` parser reads atoms by position: `rA:6CCCCCC` means 6 carbons. A molecule with `Cl` or `Br` would be `rA:2ClBr` — the first char `C` is followed by lowercase `l`, making it `Cl`.

**Why it happens:** Naive character-by-character parsing splits `Cl` into `C` and `l`.

**How to avoid:** Use the greedy two-character check: if `symbols[i+1]` is lowercase, take two characters. [VERIFIED: CONTEXT.md specifics section; AuxInfo rA: field description]

export function swatchVar(type: LayerType): string {

export function subscript(n: number): string {

}
