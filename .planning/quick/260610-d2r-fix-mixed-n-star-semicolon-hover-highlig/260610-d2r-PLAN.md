<objective>
Fix a pre-existing v1.0 hover-highlight bug: in mixed multi-fragment InChI layers of the
form `N*...;N*...` (e.g. `2*1-7-5-3-2-4-6-7;2*1-2-4-6-5-3-1`), the greedy top-level
`multMatch` regex in `ConnectionText` and `HLayerText` matches across the `;`, hijacking
control into the single-group-multiplier branch. The branch uses only `fragCounts[0]` and
n copies, so atoms in fragments after the `;` map to the WRONG canonicals (the first
fragment group instead of their own). Hovering a benzene atom highlights toluene atoms.

Purpose: Restore correct atom/bond highlighting for chemists inspecting mixed-fragment InChIs.
Output: Two surgical guards on the top-level multMatch + one regression test.
</objective>

  <action>
At both top-level multMatch sites — ConnectionText (~line 151) and HLayerText (~line 289) —
change:
  const multMatch = text.match(/^(\d+)\*([\s\S]*)$/);
to:
  const multMatch = !text.includes(';') ? text.match(/^(\d+)\*([\s\S]*)$/) : null;

This makes the single-group-multiplier branch fire ONLY for a pure `N*...` text with no
semicolon. When a `;` is present, control falls through to the existing `;`-split branch
(text.split(';'), per-segment segMult handling with cumulative offsets and fragIdx), which
already computes correct cross-fragment canonicals.

Do NOT touch ParityText or FormulaText. Do NOT touch the per-segment `segMult` regex inside
the `;`-split branches — only the two TOP-LEVEL multMatch declarations change.
