# Phase 3: InChI Display and Explanation UI - Context

**Gathered:** 2026-05-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Render the color-coded InChI layer strip, left explanation card (prose + reading-code block), and right legend card from the live Zustand store. Implement idle and layer-hover states. Include sub-token spans (formula elements, c-layer atoms, h-groups, parity markers) so they are visually present, with `setSubHover` wiring active even though Ketcher canvas highlights land in Phase 4. No mapping strip (Phase 5). No preset molecule list (Phase 5). No Ketcher canvas highlights (Phase 4).

</domain>

<decisions>
## Implementation Decisions

### Reading-Code Richness
- **D-01:** Implement **full element labels** in `readingFor()` тАФ e.g., "CтВБтАУCтВВ ┬╖ CтВБтАУCтВГ" for the c-layer and "CтВБ bears 1H" for the h-layer. This requires parsing the `rA:` field from AuxInfo to build a `atomElements: Record<number, string>` mapping (canonical index тЖТ element symbol, 1-based). Add this field to the Zustand store and extend `parseAuxMapping.ts` / `parseInchiWithAux` to populate it.
- **D-02:** Port `readingFor()` from `design_handoff_explain_that_inchi/layers-info.js` to TypeScript, replacing `mol.atoms` lookups with `atomElements` from the store. The `atomLabel(canon)` helper becomes `atomElements[canon] ?? '#' + canon`.
- **D-03:** Port `LAYER_INFO`, `DEFAULT_INFO`, and all helper functions (`formulaReading`, `parseStereoParities`, `subscript`, etc.) from `layers-info.js` to a new TypeScript module (e.g., `src/lib/layerInfo.ts`). These are final and should be ported verbatim except for module syntax and types.

### CSS Organisation
- **D-04:** Use **CSS Modules per component** тАФ one collocated `.module.css` file per new `.tsx` component, matching the established `KetcherPanel.module.css` pattern. Convert the design handoff's global class names to camelCase module references (e.g., `.inchi-layer` тЖТ `styles.inchiLayer`).
- **D-05:** CSS custom properties (tokens) from `src/styles.css` remain global тАФ accessed via `var(--c-formula)` etc. inside module files. No duplication of token definitions.
- **D-06:** Dynamic accent colours (e.g., `style={{ color: 'var(--c-conn)' }}`) are set via inline styles where the token is data-driven (layer type determines which token). CSS modules handle structural and state classes (`.active`, `.dim`, `.muted`).

### Sub-Token Spans
- **D-07:** Include `LayerText` sub-renderers in Phase 3 with full `setSubHover` wiring. Spans call `setSubHover` on mouse enter/leave; Phase 4 will act on the `subHover` store value to fire Ketcher highlights. No visual sub-token hover effect appears in Phase 3 (the store is written but nothing reads `subHover` yet) тАФ this is intentional and avoids Phase 4 needing to add event wiring retrospectively.

### Hint Text
- **D-08:** The section label hint (beside "InChI ┬╖ colour-coded layers") is **derived from the formula layer text**. When `layers` is non-empty, find `layers.find(l => l.type === 'formula')` and display its `.text` value. When `layers` is empty (canvas cleared), show nothing. No static fallback text.

### Explanation Card
- **D-09:** Use `dangerouslySetInnerHTML` for the reading-code block, matching the design handoff. All HTML strings are generated by controlled code (`readingFor()`) from parsed InChI data тАФ never from user input тАФ so XSS risk is zero. This avoids the cost of restructuring the rendering to use React components.
- **D-10:** Idle state (nothing hovered): show `DEFAULT_INFO` content тАФ title "Hover any layer", blurb from `DEFAULT_INFO.blurb`. The card has no accent color in idle state (uses `var(--ink-faint)`).

### Store Extension
- **D-11:** Add `atomElements: Record<number, string>` to the Zustand store interface and initial state (`{}`). Add `setInchiData` to accept it as a fourth argument (or update the shape of the action). `App.tsx` passes `result.atomElements` from `parseInchiWithAux`.

### Claude's Discretion
- File layout for new components (flat in `src/components/` matching existing pattern)
- Exact TypeScript types for the `readingFor()` return value
- Whether `Legend` is a standalone component or co-located in `Explanation.tsx`
- Whether `LayerText` sub-renderers are in the same file as `InchiSection` or a separate `LayerText.tsx`
- Transition timing for hover (160ms, 4px from the design handoff tokens тАФ enforce via CSS)
- Exact module class name mapping from handoff global names

</decisions>

<specifics>
## Specific Ideas

- The AuxInfo `rA:` field format (from real Ketcher 3.12.0 benzene output): `rA:6CCCCCC` тАФ the number after `rA:` is the atom count, followed by element symbols in canonical order. `atomElements` should be 1-based: `{ 1: 'C', 2: 'C', ... }`.
- The `swatchVar()` helper from the handoff (maps layer type to CSS token suffix: `c` тЖТ `conn`, `h` тЖТ `hydro`, `btms` тЖТ `stereo`) should be ported to `layerInfo.ts` and used in both `InchiSection` (layer colour) and `Legend` (swatch colour).
- Hover transition timing: `transition: opacity 160ms ease` for dim/active states; `transform: translateY(4px)` for legend tooltip slide тАФ both from the design handoff. Enforce in CSS modules.
- The legend tooltip (`.legend-tip`) in the handoff is CSS-only (`:hover` on `.legend-row` shows the nested `.legend-tip`). This is fine to port as-is; no React state needed.

</specifics>
